							Real Numbers:
									Real numbers are stored as a integer along with information showing where the radix point is. Looking at a Radix-Point
									lets say each memory location is the same size and is divided into a sign plus five decimal digits, when a variable or 
									constant is assigned the location assigned to it consists of five digits and a sign. When an integral variable or a constant 
									is assigned, the number stored in that palce is straightforward. When a real variable is declared or a real constant is defined,
									the number stored there has both a whole number part and a fraction part. The number must be coded to repersent both parts.
									Example, 
										-99999 Largest negative number
										+00000 Zero
										+99999 Largest positive number
										The precision(the maximum number of digits that can be repersented) is five digits, and each number within that range can
										be repersented exactly. What Happens when we allow oen of these digits(let's say the leftmost one) to repersent an exponent,
											example,
												+32345	repersents the number +2345*10³ (keep in mind that the leftmost digit "3" is repersenting an exponent)
												The range of numbers we can now repersent is much larger:
																			-9999*10⁹ to +9999*10⁹ 
												Now, the precision is only four digits. We can only repersent four significant digits(nonzero digits or zero digits
													that are exact) of the number itself. This means we can repersent only four-digit numbers exactly in our system.
													In Larger numbers the four leftmost digits are correct, and the balance of the digits are assumed to be zero. We
													lose the rightmost, or least significant, digits.
												example,
													Number 	Sign  Exp. 		Value
														 ↓	↓			
													+99999	 +	1   9999	+99999
												       -999999	 -	2   9999	-999999
												      +1000000   +	3   1000	+1000000
												      -4932416   -	3   4932	-4932000
																		Notice we can repersent exactly 1000000, but not -4932416. Our coding scheme
																			is limited to four significant digits; the digits we cannot repersent
																			are assumed to be zero. To extend our coding scheme to repersent real
																			numbers, we need to be able to repersent negative exponents,
																		example,
																			4394*10ˉ³= 43.94
																			or
																			22*10ˉ⁴=0.0022
																					Because our scheme does not allow for a sign for the exponent,
																					we have to change the scheme slightly. Let's let the sign that
																					we have already been using be the sign of the exponent and add a
																					sign to the left of it to be the sign of the number itself.
																					example,
																						+-99999 Largest positive number +9999*10⁹
																						↑↑
																					sign of #, sign of exponent
																						--99999 Largest negative number -9999*10ˉ⁹
																						++99999 Smallest positive number +1*10⁹
																						--90001 Smallest negative number -1*10ˉ⁹
																					Now we can repersent all of the numbers between -9999*10ˉ⁹ and 9999*10⁹
																					accurately to four digits, including all the fractional values.
																						Suppose we want to add three numbers x, y and z using this coding scheme.
																						By adding y and z, then adding x to result, The associative law of arithmetic 
																						says that the two answers should be the same-but are they?
																						example,
																							x=-1324*10³ y=1325*10³ z=5424*10⁰
																							
																							lets look at the result of adding z to the sum of x and y:
																								x+y=1*10³=1000*10⁰
																								x+y=1000*10⁰
																								  z=5424*10⁰
																									result is 6424*10⁰=(x+y)+z
																							lets look see what happens when we add x to the sum of y and z:
																								y+z=1330424*10⁰=1330*10³(truncated to four digits)
																								y+z=1330*10³
																								  x=-1324*10³
																									result is 6*10³=6000*10⁰=x+(y+z)
																									Our answers are the same in the thousands' place but are
																									different in the hundreds', tens', and ones' place, this is 
																									called repersentational error or round-off error. The result
																									of adding y to z gives us a number with seven digits of precision,
																									but only four can be stored. In addition to repersentational errors,
																									there are two other problems to watch out for in floating-point arithmetic:
																										underflow and overflow.
																									Underflow is the condition that arises when the absolute value of a calculation
																									gets to small to be repersented. for example if we had a 10­¹³ our minimum is -9,
																									therefore the result would be set to zero.
																									Overflow is the condition that arises when the absolute value of a calculation gets
																									too large to be repersented. Overflow is a more serious problem, because 10⁺²¹, our 
																									minimum is -9, so the result of the calculation cannot be stored, which we can at this
																									point issue a 10⁹ or issue a cancellation error, which can happen with floating-point
																									numbers of widely differring magnitudes when they are added or subtracted.

				Limits on communications:
								
							Parity Numbers:
									Parity bits are used to detect that an error has occured between the storing and retrieving of a byte, or the sending and recieving of a byte. 
									A parity bit is an extra bit that is associated with each byte in the hardware that uses the scheme, used to ensure that the number of 1 bits in 
										a 9-bit value(byte plus parity bit) is odd(or even) across all bytes.
										Odd parity:	requires the number of 1s in a byte plus parity bit to be odd.
										Even parity:	 requires the numbers of 1s in a byte plus parity to be even.
											Both even and odd parity use this scheme,
																if a byte contains the pattern // 11001100 //, the parity bit would be 1, giving an odd 
																number of 1s. If the pattern were // 11110001 // the parity bit would be 0 giving an odd
																number of 1s. When a byte is retrieved from memory or recieved from a transmission, the 
																number f 1 bits is counted(including the parity bit). If the number is even, an error has
																occurred. If the scheme is used in the hardware each byte actually has an extra bit,
																accessible only by the hardware, that is used for error detection.

										walk-through the team performs a manual simualtion of the deisgn or program with sample test inputs, keeping track of the programmers data by hand
